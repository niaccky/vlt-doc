Virtual Log TraceA visual log tracing reference implementation for multi-source systems

It combines a protobuf-first protocol, one ingestion endpoint, and a Trace Workbench so logs from services, jobs, functions, and message queues can be searched, grouped, and explored as a trace.

Start in 5 Minutes

Protocol-first ingestion

Ingest batches through `POST /v1/logs`, with support for `application/protobuf`, `application/protobuf+json`, `application/json`, and `gzip`.

Query and aggregation

Search logs, then move to trace detail, timeline lanes, topology edges, and grouped source inventory.

End-to-end reference stack

The repository already contains the Go server, a Go sender, a React workbench UI, and multiple sample payloads.

This documentation focuses on three things: what the project is, what it can do, and which behaviors are actually implemented today.

SenderUse the bundled Go sender or your own client to emit `ingest.v1.UploadLogsRequest` batches.
Ingest and validationThe server performs token authorization, batch validation, record rejection, idempotency caching, and source/body normalization.
Query layerThe in-memory store serves log search, trace detail, timeline lanes, topology edges, and grouped sources.
Trace WorkbenchThe frontend links logs, topology, timeline, and inspection into one operator-facing flow.
Sender → Ingest API → Normalized Records → Query API → Workbench

What you can use it for ​

  • Define a stable upload protocol instead of scattering log shape across multiple systems.
  • Use trace_id, source_name, and instance_id to turn multi-source logs into trace-shaped data.
  • Demo the full path from “upload a batch” to “inspect timeline and topology in the UI”.
  • Prototype the protocol and workbench behavior for a future observability product.

Who this documentation is for ​

  • Engineers onboarding into the virtual-log-trace repository
  • Clients integrating with the upload protocol
  • Frontend or platform engineers validating query payloads and aggregation semantics
  • Anyone using the current implementation as a starting point for further product work

A 5-minute path ​

  1. Start the backend in the virtual-log-trace repository root: go run ./cmd/vlt-server
  2. Ingest sample logs: go run ./cmd/vlt-sender -token dev-token -project demo -environment dev -format json
  3. Start the frontend from web: npm install && npm run dev
  4. Or use the all-in-one flow: ./scripts/dev-preview.sh or make preview
